# 教程 07: 编码与加密算法知识

## 概述

在渗透测试和越权漏洞挖掘中，攻击者经常遇到对参数进行了编码或加密的 API 接口。理解这些算法的原理和特征，能够帮助你快速识别编码类型并解码参数，从而发现潜在的越权漏洞。

本教程介绍常见的编码、加密和签名算法概念，以及它们在真实业务系统中的典型应用场景。

---

## 编码 vs 加密 vs 签名

| 类型 | 特点 | 典型算法 |
|------|------|----------|
| **编码 (Encoding)** | 可逆变换，无密钥，仅改变数据表示形式 | Base64, Base32, Base58, Hex |
| **加密 (Encryption)** | 需要密钥，提供机密性保护 | AES, RSA, SM2, SM4 |
| **签名 (Signature)** | 使用密钥生成验证码，确保数据完整性 | HMAC-SHA256, MD5+SALT |

---

## 常见编码算法

### Base64

最常用的二进制到文本编码方案，将任意字节序列转换为 A-Z、a-z、0-9、+、/ 共 64 个可打印字符，末尾以 `=` 填充。

- 特征：包含大小写字母、数字、`+`、`/` 和结尾 `=` 
- 识别：`^[A-Za-z0-9+/]+=*$`

```
原始:  hello
Base64:  aGVsbG8=
```

### Base32

使用 A-Z 和 2-7 共 32 个字符，常用于需要大小写不敏感的场景。

- 特征：仅包含 A-Z 和 2-7，可选 `=` 填充
- 识别：`^[A-Z2-7]+=*$`

### Base58 (Bitcoin 风格)

去除了容易混淆的字符（0、O、I、l），常用于加密货币地址。

- 特征：字母数字混合，无 `0`、`O`、`I`、`l`
- 识别：`^[1-9A-HJ-NP-Za-km-z]+$`

### Hex (十六进制)

每个字节用两个十六进制字符表示。

- 特征：仅包含 0-9 和 A-F
- 识别：`^[0-9A-Fa-f]+$`

### 凯撒密码 (Caesar Cipher)

最经典的单表替换密码，每个字母按固定偏移量移位。对数字同样适用。

- 偏移量常见为 3（ROT3）
- 字母：A→D, B→E, ..., Z→C
- 数字：1→4, 2→5, ..., 7→0, 8→1, ...

### 自定义编码表

标准编码算法可以通过修改字符表实现混淆。例如：
- 标准 Base64 字符表：`ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/`
- 自定义变体：反转字符表顺序，或任意排列

识别方法：对比已知明文的标准编码输出与目标系统的输出，找出字符映射关系。

### 多层嵌套编码

参数可能经过多次编码，例如 Base64 → Base32 双层嵌套。解码时需反向逐层剥离。

## 常见加密算法

### AES (Advanced Encryption Standard)

对称加密标准，广泛应用于数据传输和存储加密。

- **AES-256-GCM**: 256 位密钥，GCM 模式提供认证加密
- 密钥长度：128 / 192 / 256 位
- 加密结果通常以 Base64 编码传输
- 需要 Initialization Vector (IV) 或 Nonce

### RSA

非对称加密算法，使用公钥加密、私钥解密。

- **RSA-OAEP**: 推荐的填充方案，安全性高于 PKCS#1 v1.5
- 密钥长度通常为 2048 或 4096 位
- 适用于小数据量加密和密钥交换

### SM2 / SM4 (中国国家密码标准)

- **SM2**: 椭圆曲线非对称加密，对标 ECC
- **SM4**: 对称分组加密，对标 AES
- SM4-CBC 模式需要 IV 和 PKCS7 填充

## 签名与完整性校验

### MD5 / SHA 哈希签名

部分 API 使用"参数值 + SALT"计算哈希作为签名，防止参数被篡改。

```
格式: value:md5(value + salt)
```

问题：如果 SALT 固定且可预测，攻击者可重新计算哈希伪造参数。

### HMAC (Hash-based Message Authentication Code)

使用密钥和哈希函数生成消息认证码。

- **HMAC-SHA256**: 使用 SHA256 的 HMAC
- 签名计算：`HMAC(key, payload)`
- 密钥泄露将导致任意参数可被伪造签名

### 真实业务中的签名机制

生产环境中，签名通常作为 HTTP 请求头传递：

```
X-Signature: <hash_value>
X-HMAC-Sign: <hmac_value>
X-Timestamp: <unix_timestamp>
```

签名原始数据通常为请求参数的规范化字符串（如排序后的 `key=value` 对）或 JSON body。

## 攻击者视角：如何识别和破解

1. **观察参数模式**：检查参数是否包含 Base64/Base32/Hex 特征字符
2. **对比已知明文**：如果知道某个参数的真实值（如自己的用户 ID），对比编码值推断算法
3. **尝试标准解码**：不成功则考虑自定义编码表或加密
4. **寻找密钥泄露**：检查前端源码、公共接口、错误信息中的密钥信息
5. **分析签名机制**：尝试修改参数观察签名验证失败信息，推断签名算法

---

## 密码学在安全防护中的正确使用

- **加密**：保护数据机密性，但不能替代访问控制
- **签名**：确保数据完整性，但不能替代权限校验
- **编码**：仅改变数据格式，不提供任何安全保护
- **安全根本**：后端权限验证是所有安全机制的基础
